Work can move faster than understanding.

Start with [Integration Lag](/en-us/signals/operational/work-delivery/integration-lag/):

work can move faster than shared understanding.

The next question is:

> what can we do once we see it?

Not a system to adopt.

Not a process to roll out.

A way to keep seeing clearly while work is moving.

### Where it starts

It often starts with a feeling:

> something is off

The work may look fine from the outside.

But underneath it, people notice:

- more rework
- unclear ownership
- explanations that no longer hold
- decisions that are hard to trace
- confidence dropping while output rises

The first move is to name what is happening.

### Name the lag

Integration lag is the gap between output speed and shared understanding.

The phrase helps because it moves the problem away from blame.

It is not only that someone missed something.

It is that the system is producing faster than the team can understand together.

### Ground in reality

The next move is grounding.

Do not begin with what the documentation says, what people remember, or what AI
summarizes.

Begin with what is actually present.

That means looking at:

- code
- tests
- docs
- decisions
- workflows
- current behavior

This is forensic onboarding.

It means replacing assumed understanding with observable reality.

### See misalignment

Once you look closely, gaps become visible.

For example:

- docs say one thing, but the code does another
- tests cover less than people think
- decisions exist, but their reasons are hard to find
- the way AI is guided does not match how the repository actually works

These gaps are not just annoying.

They are risk signals.

### Maintain understanding

Understanding does not maintain itself.

It fades when people leave, work speeds up, docs get stale, or decisions are
made under pressure.

So understanding needs a place to live.

That place might be:

- a pull request
- an architecture note
- a test
- a validation script
- a source-of-truth map

The goal is not to document everything.

The goal is to stop forcing people to carry repeatable knowledge in their heads.

### Let constraints emerge

Good constraints come from the system itself.

They come from what repeatedly breaks, hides work, or creates rework.

A useful constraint might become:

- a checklist
- a rule
- a test
- a validation check
- a clearer review path

The constraint should reduce confusion.

If it only adds pressure, it is not helping.

### Use AI differently

AI can generate more output.

But that is not the important shift here.

AI can also help inspect:

- what changed
- what conflicts
- what is missing
- what needs checking

The shift is:

> less generation, more visibility

### Keep the loop alive

The path is a loop:

1. inspect what is real
2. find the mismatch
3. name the risk
4. decide where the learning belongs
5. check whether it helped
6. remove what is stale
7. repeat

This is how integration lag gets smaller.

Not by moving slower for its own sake.

By making understanding part of the work.

### What comes next

If the work needs a ready way to try the practice, go to
[Sensible Defaults](/en-us/core-system/mandate-lenses/sensible-defaults-a-lens-you-can-load/).

If the work still feels unclear, the next practical step is forensic onboarding.

Understand what is present before trying to change it.

### Closing

The future of AI-assisted delivery is not only better generation.

It is better integration.

Teams do not only need more output.

They need to keep understanding what the output means.
