We hit a point where velocity went up, but coherence went down.

AI generated code. Tickets moved faster. Delivery pipelines flowed.

Everything looked efficient.  
But the understanding behind it was thinning.

And still, something stopped adding up.

We kept shipping.  
We were no longer sure what we were shipping into.

### Integration lag

The bottleneck is no longer production.

It is integration.

Not CI/CD integration.

Integration of understanding:

- what a change actually means
- what assumptions it relies on
- what work is hidden behind it
- what it might break

AI made generation cheap.

It also made it easy to move faster than our ability to inspect, review, and make sense of what we are doing.

This is what we call **integration lag**.

Agreement was visible.  
Understanding was not.

### How it shows up

This does not look like failure.

It looks like pressure:

- features that "worked" collide in production
- dependencies surface too late
- teams discover missing work after commitment
- rework happens under time pressure
- fixes move into nights or weekends

Nothing obviously broken early on.  
The cost appears later.

### What we changed

We did not introduce a framework.

We changed how we make decisions before committing work.

Before committing, we made reasoning visible.

Not as documentation.  
Not as a requirement.

As a working intervention while the decision is still forming.

### The intervention

We use a source context that can guide how an LLM or agent reasons during work.

That context is not only a prompt. It relies on the seed contract behind the
practice, the lens text, and the context seeder that carries the relevant parts
of that contract into use.

It makes reasoning surface while the work is still forming.

In practice:

- the agent does not jump straight to output
- it surfaces assumptions
- it identifies missing domains
- it exposes hidden work
- it keeps the next check visible

We call this lens **Sensible Defaults**.

It is not a framework.

It is the first public approach to runtime reasoning guidance for LLM-assisted
coding.

### Example

Before:

> "Implement feature X with API Y"

Looks complete. Moves fast.

After running it through the lens:

- data ownership unclear
- security implications missing
- deployment dependencies not considered
- testing scope undefined

Same task.

Rework surfaced before it happens.

### What this actually does

It may feel slower at first.

It shifts effort forward, instead of paying for it later as rework.

It turns:

- implicit -> visible
- assumed -> inspectable
- aligned -> understood

This is not about getting more out of AI.

It is about constraining AI so it does not outrun understanding.

### Use it

Don't roll this out.

Try it once.

Take one real, slightly uncomfortable task.

Use the Sensible Defaults source context.  
Run the task through the lens.

See what assumptions, missing domains, and next checks surface.

If it doesn't reduce rework or late surprises, drop it.

### This is not a framework

If this turns into process, it fails.

If it becomes a requirement, it fails.

If it becomes something to complete, it fails.

If it is used to control people instead of exposing reality, it fails.

This only works as a practice.

### This should not stay private

Reasoning can begin with one person and an AI system.

But if the work affects other people, the reasoning cannot stay private.

It has to become visible enough for others to inspect, question, and carry
forward.

Not installed.  
Practiced with trace.

### The real goal

Not better output.

Better understanding.

Because under AI speed, that is one of the few things that keeps systems coherent.

Everything else is just faster confusion.
