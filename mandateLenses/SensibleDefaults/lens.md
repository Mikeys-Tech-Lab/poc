## What this lens is for

Software delivery breaks when planning treats delivery as implementation only.

This lens exists to make hidden delivery work visible early, especially when
people are already under pressure and the system around them is unclear.

It is for situations where:

- output is moving faster than shared understanding
- plans compress discovery, risk, and coordination into "build time"
- cognitive overhead is rising because the real shape of the work is invisible
- legacy systems or chaotic project conditions make hidden work accumulate

This is a planning-visibility aid.

It is not:

- a scoring system
- a compliance gate
- a way to judge people
- a reason to demand exhaustive documentation

---

## Core defaults

When this lens is active, assume:

1. Implementation effort is only one part of delivery effort.
2. Shared understanding precedes solutioning.
3. Risks, assumptions, issues, and dependencies must be named early.
4. Working outcomes should appear early and evolve iteratively.
5. Security and resilience start at the beginning, not at the end.
6. Local decisions must stay aligned with system intent.

These defaults are intentionally small.
If the lens grows by trying to cover everything, it has drifted.

## Minimal intervention questions

Before committing to scope, sequencing, or dates, ask:

- What outcome is being created, and how will it be checked?
- Which delivery domains are active, and which are unaccounted for?
- What is the next check that reduces uncertainty early?

## Interpretation rule

When reading a plan, request, estimate, or mandate under this lens, verify:

- which domains are clearly active
- which domains are likely required but missing
- which risks are hidden or postponed
- which coordination work is absent
- whether the timeline reflects systemic effort instead of coding time only

Treat missing domain acknowledgement as a delivery risk signal, not a neutral
omission.

Exposure levels:

- `routine` - domain is acknowledged and addressed
- `elevated` - domain is acknowledged but deferred or unclear
- `critical` - domain is likely required but absent

---

## Default output shape

When the lens is explicitly invoked, output:

1. **Active domains** - what is already accounted for
2. **Missing domains** - what is probably required but not planned
3. **Often unplanned work** - concrete work that will still happen
4. **Risk concentrations** - where reality is most likely to punish optimism
5. **Timeline pressure points** - where sequencing or dependencies will bite
6. **Suggested next questions** - the smallest clarifications that reduce risk

Sizing rule:

- for small changes, keep each section short
- if a domain is not applicable, say so briefly
- do not inflate uncertainty into narrative; reduce it into the next honest
  check

---

## Misuse guard

This lens is being misused if it becomes:

- compliance scoring, surveillance, or performance evaluation
- a blame tool
- a demand for exhaustive paperwork on low-stakes work
- a way to turn uncertainty into a performance of certainty

If it adds pressure instead of releasing it, it has drifted.
